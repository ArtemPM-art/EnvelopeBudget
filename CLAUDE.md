# EnvelopeBudget — памятка проекта

Этот файл читается автоматически при каждом запуске. Он описывает продукт, роли, стек и — главное — **нестандартную инфраструктуру сборки**. Прочитай раздел «Инфраструктура» до того, как трогать код: здесь нет Mac и нет Xcode, и это меняет почти всё.

---

## 1. Продукт

Нативное iOS-приложение для личного учёта финансов по **конвертному методу**.

**Ценностное предложение:** пользователь вводит трату **в момент покупки** и сразу видит остаток конверта по категории. Это снимает тревогу за деньги и делает финансы прозрачными.

**Ключевая идея модели:** деньги делятся **по назначению** (конверты: еда, транспорт, развлечения), а **не по месту хранения** (карта 1, карта 2, наличные).

**Пользователь ровно один — сам владелец проекта.** Приложение не пойдёт в App Store, у него не будет чужих пользователей. Это не оправдание для плохого кода, но это законный аргумент при выборе минимальной версии iOS и при отказе от поддержки старых устройств.

Целевое устройство: **iPhone 12 mini, портретная ориентация** (375×812 pt). Макеты проверяются под него.

## 4. Инфраструктура сборки — ЧИТАТЬ ОБЯЗАТЕЛЬНО

**У PO нет Mac. Он на Windows. Xcode не существует в этом проекте.**

Отсюда следуют жёсткие правила:

### Как проект собирается

```
Правка кода (локально, Windows)
  → git push в main
  → GitHub Actions поднимает macOS-раннер
  → XcodeGen генерирует .xcodeproj из project.yml
  → xcodebuild собирает БЕЗ подписи (CODE_SIGNING_ALLOWED=NO)
  → .ipa публикуется в Releases
  → PO скачивает и ставит на iPhone через AltStore (бесплатный Apple ID)
```

### Что это означает на практике

- **Нет симулятора. Нет превью. Нет проверки компиляции локально.**
  Ты не можешь запустить код и посмотреть. Единственная обратная связь — лог GitHub Actions через 5–10 минут.
- **Поэтому код должен компилироваться с первого раза.** Пиши осторожно, проверяй импорты, не полагайся на «потом поправим». Каждая опечатка стоит 10 минут.
- **`.xcodeproj` в репозитории НЕТ и быть не должно.** Единственный источник правды о структуре проекта — **`project.yml`** (XcodeGen). Добавил новый файл в `Sources/` — он подхватится автоматически, править `project.yml` для этого не нужно. Менять `project.yml` нужно только при изменении настроек сборки, Info.plist, зависимостей.
- **Каждый push в `main` запускает сборку.** Репозиторий публичный, macOS-минуты бесплатны и без лимита. Но не оставляй `main` в сломанном состоянии.

### Что НЕЛЬЗЯ трогать без явного разрешения PO

- **`PRODUCT_BUNDLE_IDENTIFIER` = `com.artem.envelopebudget`.**
  Бесплатный Apple ID даёт лимит на уникальные идентификаторы (10 за 7 дней). Пока идентификатор один — переустанавливать сборки можно бесконечно. Сменишь — начнёшь жечь лимит.
- **Три флага отключения подписи** в `build.yml` (`CODE_SIGN_IDENTITY=""`, `CODE_SIGNING_REQUIRED=NO`, `CODE_SIGNING_ALLOWED=NO`). Без них сборка упадёт: платного аккаунта Apple нет, подписывать в CI нечем. Подпись ставит AltStore на стороне PO.
- **Обязательные ключи в Info.plist** (`LSRequiresIPhoneOS`, `CFBundleExecutable`, `CFBundleIdentifier`, `CFBundlePackageType` и т.д.). Их отсутствие приводит к тому, что `.ipa` собирается, но не устанавливается. Это уже стоило проекту нескольких часов отладки — не повторяй.

### Бытовое ограничение

Приложение подписано бесплатным Apple ID и **перестаёт открываться через 7 дней**, пока PO не переподпишет его через AltStore. Это известно и принято. Платный аккаунт Apple ($99/год) отклонён.

## 5. Правила кода

- Код на продакшн, не на демо. Не заготовки, не скелеты — рабочий код.
- **Никаких `try!` и force unwrap (`!`)** без явного обоснования. Обработка ошибок — всегда.
- Один компонент — один файл. Не складывать всё в `ContentView`.
- Комментарии — только там, где логика неочевидна. Не комментировать каждую строку.
- Именование — на английском, понятное, без сокращений.
- Тексты интерфейса — **на русском**. Валюта — **рубли (₽)**, суммы в целых рублях, копейки продукту не нужны.

## 6. Как общаться с PO

- Объяснения **на языке продукта**, не на языке кода. Технические детали — только если PO спрашивает или они критично влияют на продуктовое решение.
- Технические термины объясняй при первом употреблении.
- Оценки сложности — в понятных единицах («простая задача — 30 минут», «сложная — несколько часов»), не в story points.
- Если задача требует значительного рефакторинга — **предупреждай до того, как выдаёшь код**, не после.
- После кода — **инструкция «как проверить»**: что PO должен нажать и увидеть, чтобы убедиться, что задача выполнена. Проверка идёт по пунктам AC.
- Если что-то в текущей стори физически непроверяемо (например, «потрачено» всегда 0, потому что ввода трат ещё нет) — **скажи об этом заранее**, чтобы PO не принял это за баг.

## 7. Принятые продуктовые решения (не пересматривать самовольно)

Из discovery, проведённого PO на себе за 5 недель жизни с конвертным методом в Google Sheets.

### Граница осознанности

Осознанность создаёт ценность (снимает тревогу) — её **не автоматизируем**. Рутину — автоматизируем.

| Вручную (осознанность) | Автоматом (рутина) |
|---|---|
| Сумма траты | Дата (по умолчанию «сейчас») |
| Категория (конверт) | Тип операции (по умолчанию «расход») |

Осознанность живёт на уровне **категории и остатка конверта** («на еду осталось 3000»), а **не** на уровне отдельных позиций («печенье 89 + молоко 70»). Поэтому названия товаров не вводятся — только конверт и сумма.

### Типы конвертов

- **Расходные** — рутина месяца (еда, транспорт). Участвуют в месячном балансе.
- **Накопительные** — копилка. Трата из них не ломает месячный баланс.

### Честность при перерасходе

Остаток уходит **в минус честно и показывается красным** (доступно 20000, потратил 21000 → «−1000»). Не прятать минус, не обнулять. Честность важнее комфорта — это ядро продукта.

Перерасход покрывается **выбором конверта-донора** пользователем (категории разной природы: буферы, фиксированные, гибкие — жёсткий перенос внутри той же категории не работает).

### Событие «поход в магазин»

Пользователь запускает событие одной кнопкой → все траты попадают в один контейнер, суммируются по категориям → отражаются в конвертах **в режиме черновика** → пользователь может отказаться и всё откатится → закрывает событие одной кнопкой, траты учитываются окончательно. Событие опционально.

Внутри события работает **принцип калькулятора**: ввёл цену, выбрал категорию — сумма прибавляется к уже введённым по той же категории.

## 8. Что сознательно НЕ делаем

- **Балансы счетов и карт** — чужая модель. Конвертной модели не нужно знать, на какой карте лежат деньги.
- **Интеграция с банками** — рудимент постфактумного учёта. При вводе в момент покупки не нужна.
- **Бэкенд, синхронизация, аккаунты** — не нужны. Всё локально.
- **Автораскладка бюджета по конвертам за пользователя** — убивает осознанность. Отложено.

**Фильтр для любой новой фичи:** «Она работает в конвертной модели (деньги по назначению) или тащит в учёт по счетам (деньги по месту хранения)?» Если тащит к счетам — почти наверняка не наш продукт.

## 9. Roadmap

- **Now:** ввод траты в момент покупки с остатком конверта; конвертный метод (расходные + накопительные конверты).
- **Next:** ускорение выбора категории; раскладка бюджета по конвертам (ползунки); rollover между месяцами.
- **Later:** автораскладка; аналитика; детализация по позициям через скан чека.

## 10. Текущее состояние

Труба сборки доказана: пустое приложение успешно собралось на GitHub Actions и запустилось на реальном iPhone через AltStore.

Реализуется **стори 1 — создание конверта**:
создание конверта с названием и суммой, отдельный экран списка, отдельный экран редактирования по тапу, неограниченное число конвертов, удаление, запрет дубликатов названий.

Следующая стори — ввод трат и «поход в магазин». Она принесёт с собой сущность «нераспределённые траты», которой пока нет.

**Важно:** пока ввода трат нет, у любого конверта «потрачено» = 0, а «осталось» = «заложено». Это не баг.
